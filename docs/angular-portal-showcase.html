Architecture principles

16 nguyên tắc cấp cao quyết định code thật mà các skill sinh ra. Source of truth tại skills/tracks/angular-portal/_refs/architecture-principles.md. Tóm tắt 6 nguyên tắc chính bên dưới — đọc full file cho 16 principles + anti-principles.

1. Feature-first folder structure

src/libs/<module>/features/<entity>/ — xoá feature = xoá 1 folder. Bug trong product = grep 1 folder. Không có module-by-type kiểu controllers/, services/, views/.

3. No cross-module imports (killer rule)

Module A KHÔNG import từ Module B. Cả 2 cùng import từ @sd-angular/core hoặc src/shared/. Enforced bởi eslint-plugin-boundaries + review flag Critical violation.

4. UI dumb, services own logic

Component: wire input → call service, display state, navigate. Service: business rules, derived data, caching. Method >50 dòng trong service = split. Method >15 dòng trong component = push xuống service.

5. Signal-first, RxJS chỉ ở boundary

signal() mutable / computed() derived / effect() side-effect. RxJS chỉ tại HTTP + form valueChanges + 3rd-party APIs. BehaviorSubject → signal, combineLatest → computed.

9. Bốn layout chuẩn (không phát minh)

side-drawer / UnifiedCompact / UnifiedSplit / AdaptiveSplitDetail. Screen không vừa = đang giải sai bài toán. Custom layout phải mark // CUSTOM_LAYOUT: <reason> + flag for design review.

16. Portal ≠ Agent

Skill set này sinh ra Angular portals. 16 principles này quy định code SINH RA, không phải code của agent. Khi principle đổi → propagate sang skill emit (11/12-init-*) + reviewer (50-review-code) + auto-docs template.

Golden path — anatomy of one entity

Một entity hoàn chỉnh (vd product trong module catalog) sinh ra tree dưới đây. Code đầy đủ ở _refs/templates/example-product.md.

src/libs/catalog/
├── catalog.configuration.ts                        # InjectionToken + interface
├── catalog.module.ts                                # NgModule (hybrid mode only)
├── routes.ts                                        # lazy-load children + module-level guard
├── guards/catalog.guard.ts                          # reads route data.permission
└── features/product/
    ├── product.routes.ts                            # data: { permission: 'CRM_PRODUCT_LIST' }
    ├── pages/list/
    │   ├── list.component.ts                        # SdTable + external filters + 4 audit cols
    │   ├── list.component.html
    │   └── list.component.spec.ts                   # permission + filter + sort tests
    ├── pages/detail/
    │   ├── detail.component.ts                      # 3-state CREATE/UPDATE/DETAIL
    │   ├── detail.component.html
    │   └── detail.component.spec.ts                 # state transitions + save-flow tests
    ├── components/
    │   └── product-select/                          # entity-internal autocomplete
    └── services/
        ├── product.model.ts                         # ProductSaveReq + ProductDTO

Testing strategy

Coverage approach (chốt ở clarify)

post-hoc (default cho CRUD scaffolding) — code trước, test sau. TDD (default cho validators / workflows / route guards) — test bones trước, implement đến khi pass.

Borrowed từ superpowers:test-driven-development.

Coverage levels

minimal: chỉ should create. standard (default): + permission + data visibility + sort + form save-flow. full: + edge cases + state transitions.

Người dùng chọn ở sdcorejs-clarify-requirements.

Test pyramid (theo philosophy.md)

Unit: pure functions / validators / mappers / single-service-method. Integration: TestBed + HttpClientTestingModule + 2-3 collaborators. E2E: Cypress hoặc Playwright (auto-detect).

Mocking strategy

Services: MockCrudStore localStorage (production code itself; no extra mocking). Components: TestBed mocks cho Router, ActivatedRoute, services. Templates: TestBed.overrideComponent để isolate khi Core UI deps quá nặng.

Core conventions enforced

Core UI first

Mọi screen / form / table dùng @sd-angular/core components nếu có (SdTable, SdSection, SdAnchorV2, SdSideDrawer, SdUploadFile…). Custom UI chỉ khi không có equivalent — và phải mark // CUSTOM_UI: <reason> + alert TODO stub.

Mock-first services

Default MockCrudStore via localStorage cho mọi entity service. 20-40 domain-realistic seed rows (không "Name 1" / "Code 01"). Switch sang BaseService chỉ khi backend contract explicit.

Signal-first state

signal() cho mutable state (loading, entity, tableOption, state machine). computed() cho derived. effect() chỉ cho side-effects. Templates dùng @let hoặc computed nếu signal ≥2 lần.

inject() not constructor

readonly #service = inject(MyService) trên mọi DI. KHÔNG dùng constructor injection. #private prefix cho mọi field nội bộ. changeDetection: ChangeDetectionStrategy.OnPush mặc định.

4 detail layout variants

side-drawer cho ≤6 fields, 1 section, no workflow. UnifiedCompact cho medium form. UnifiedSplit khi cần title context. AdaptiveSplitDetail cho workflow heavy + multi-section (DETAIL khác CREATE/UPDATE).

Permission codes

Module → Entity → Action order. UPPERCASE. data: { permission: 'CRM_PRODUCT_LIST' } trên mọi route. Module guard tự đọc — KHÔNG manual canViewList check trong component.

4 audit columns

Primary list pages luôn có createdAt, createdBy, updatedAt, updatedBy ở cuối. Type datetime + string, widths 180px. Skip cho sub-tables, lookup entities, dialog tables.

Bilingual VI/EN

VI request → VI output (full diacritics cho labels/messages). EN → EN. Permission codes + route paths stay English. Error messages bilingual { vi, en } shape.

Workflow action visibility

State-driven AND permission-driven. Button "Approve" chỉ show khi state===PENDING AND user có permission APPROVE. Bulk actions on list cũng theo rule này.

Superpowers integration

Pipeline borrows từ anthropic-skills:superpowers để giữ discipline:

Brainstorming as MUST

Auto-route từ clarify khi scope mơ hồ. Cheap brainstorm > expensive mis-targeted code.

Writing plans

Numbered file-by-file plan với verification commands, user-approval gate trước khi code.

TDD option

Clarify hỏi post-hoc hay TDD. TDD shifts test bones BEFORE code (validators, workflows, route guards).

Verification before completion

verify-before-done block "done" cho đến khi mỗi acceptance criterion ✅ verified.

Receiving code review

repair-loop verify finding correctness trước khi fix (drop stale / mis-scoped / redundant). Không apply mù.

Finishing development branch

branch-ready sweep debug logs / secrets / focused tests / lint pass trước khi commit.

Subagent-driven development

parallel-dispatch + subagent-driven-dev cho fan-out ≥3 independent units.

Systematic debugging

sdcorejs-debug enforce reproduce → isolate → hypothesize → root-cause cho non-trivial bugs.
