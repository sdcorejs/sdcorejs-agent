# 22 — Screen: CREATE state

## Purpose
Implement the CREATE branch of an entity's `pages/detail/detail.component.ts`. This is the empty-form state used when the user hits `/create` on the entity. The component shell (template, imports, signals, navigation helpers) is owned by `21-screen-detail.md`; this skill only adds CREATE-specific rules.

## When to use
- User asks "tạo màn create cho ...", "create page", "form tạo mới", "new record screen"
- Generating a fresh entity and the CREATE state has not been wired yet
- Refining an existing detail component that does not yet handle the CREATE branch correctly (no `state` signal, missing submit, wrong navigation)
- After `12-init-entity` if create-specific rules need extra attention (file uploads pre-save, default values, parent-id from query param)

If the user is asking about read-only DETAIL or about the editable UPDATE state, defer to `21-screen-detail.md` or `23-screen-update.md`.

## Reference: shared component shell
Read `21-screen-detail.md` first for:
- file path (`pages/detail/detail.component.ts`)
- `ViewState = 'CREATE' | 'UPDATE' | 'DETAIL'` discriminator
- imports of `SdInput / SdSelect / SdDate / SdTextarea`
- the FormGroup definition (do not redefine here)
- `pageTitle`, `headerRight` button structure
- the `onBack()` and `loadEntityData()` flow

This skill assumes that file already exists (or is being generated by `12-init-entity`).

## CREATE-state rules

### Route
`{ path: 'create', loadComponent: () => import('./pages/detail/detail.component').then(m => m.DetailComponent), data: { permission: '<MODULE>_C_<ENTITY>_CREATE', permissionKey: '<module>' } }`

### State detection
```typescript
ngOnInit() {
  const id = this.#route.snapshot.paramMap.get('id');
  if (!id) {
    this.state.set('CREATE');
    this.form.enable();
    this.pageTitle.set(`Tạo mới ${ENTITY_LABEL}`); // VI portal
    this.applyDefaults();   // see below
    return;
  }
  // ... UPDATE / DETAIL branches handled in 23-screen-update.md / 21-screen-detail.md
}
```

### Form behavior
- Form is empty by default; do **not** call `service.detail(id)`
- Apply default values where domain dictates:
  - `isActivated: true`
  - `status: 'DRAFT'` (or domain default)
  - parent id from `route.snapshot.queryParamMap.get('parentId')` when scaffolding child entities
- All fields editable, no `[viewed]` flags

### File uploads
If the form has `sd-upload-file` controls:
1. `await Promise.all(this.uploadFiles().map(f => f.upload()))` BEFORE calling `service.create(...)`
2. Patch returned `fileIds` into the form value
3. Then submit

### Submit
```typescript
async onSave(): Promise<void> {
  if (this.form.invalid) {
    this.form.markAllAsTouched();
    this.#notify.error('Vui lòng kiểm tra các trường bắt buộc');
    return;
  }
  this.saving.set(true);
  try {
    this.#loading.show();
    if (this.uploadFiles().length) {
      await Promise.all(this.uploadFiles().map(f => f.upload()));
    }
    const payload = this.form.getRawValue() as EntitySaveReq;
    const created = await this.#service.create(payload);
    this.#notify.success('Tạo mới thành công');
    // Navigate back to list (default) OR to detail of the new record (opt-in)
    this.#router.navigate(['..'], { relativeTo: this.#route });
  } catch (err) {
    this.#notify.error('Tạo mới thất bại');
  } finally {
    this.saving.set(false);
    this.#loading.hide();
  }
}
```

### Navigation after success
Default: navigate up one level (`['..']`) → list page. Alternative (opt-in when user explicitly asks): navigate to `['../detail', created.id]`. Document the choice in the auto-docs skill output.

### Permission gate
Save button uses `*sdPermission="'<MODULE>_C_<ENTITY>_CREATE'; sdPermissionKey: '<module>'"`.

### Workflow integration
If entity has workflow (see `31-workflow-actions.md`), CREATE state may show two submit variants:
- `Lưu` → calls `service.create(...)`
- `Lưu & Gửi duyệt` → calls `service.create(...)` then `service.submit(id)`
Both go through the same validation gate (`form.invalid → markAllAsTouched`).

## Anti-patterns
- Calling `service.detail(id)` in CREATE — there is no id yet
- Using `[viewed]="true"` on form fields in CREATE state
- Showing approve/reject/edit buttons in CREATE
- Submitting without `markAllAsTouched()` on invalid form (UX leaves errors hidden)
- Uploading files AFTER `service.create(...)` (orphans the file blobs if create fails)
- Hard-coding navigation target without considering the user's explicit instruction
- Forgetting `data.permission` on the `/create` route
- Skipping the loading spinner / notify on save failure
